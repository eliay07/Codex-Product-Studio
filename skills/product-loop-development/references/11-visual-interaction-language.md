# 非文字化 UI 表达与游戏化指引

## 一句话原则

成熟产品不要用大段文字解释流程，而要让用户通过图标、位置、颜色、状态、动效和控件形态“看一眼就知道下一步”。

## 用户偏好

- 喜欢游戏化一点的界面，但不是幼稚卡通。
- 主要流程可以用 tab、步骤、地图、进度轨道、任务卡、徽章、状态灯表达。
- 每个主要 tab 或步骤都应有图标或视觉符号，不只靠文字。
- 用户不应该读很多说明才知道怎么做。
- 页面语言要短、准、像成熟产品，不要像 PRD。
- 解释型长句、产品策略、实现意图不能直接写在页面上。

## UX 依据

- NN/g 的“Recognition rather than recall”原则强调：界面应让用户识别信息，而不是强迫用户记忆或阅读长说明。
- Material Design 认为 icon button 适合紧凑工具动作，但需要清晰语义；tooltip 应简短、具体、面向动作。
- Apple HIG 的 tab bar / icon 指南强调图标可帮助用户识别内容或功能，但图标和标签要稳定一致。

## 文案压缩规则

坏例子:

```text
先把这条内容做给谁、做成什么说清楚
项目页只服务一件事，就是把受众、目标、表达角度、交付和发布时间锁定。
```

成熟产品写法:

```text
内容定位
```

或:

```text
定位
受众
角度
交付
排期
```

更好的 UI 表达:

```text
[靶心图标] 定位
[人群图标] 受众
[罗盘图标] 角度
[包裹图标] 交付
[日历图标] 排期
```

## 页面标题规则

页面标题只写对象或任务名:

```text
内容定位
项目详情
发布计划
素材库
选题池
```

不要写:

```text
先把...
本页面用于...
我们要...
你需要在这里...
项目页只服务...
```

如果确实需要解释，放到:

- tooltip
- onboarding 第一次提示
- 右侧帮助抽屉
- 空状态短提示
- PRD / 设计说明 / 交付说明

## 图标化步骤模型

复杂流程优先做成视觉步骤，而不是长段文字:

```text
1 定位 -> 2 素材 -> 3 草稿 -> 4 审核 -> 5 发布 -> 6 复盘
```

每步至少有:

```text
图标
短标签
状态: 未开始 / 进行中 / 已完成 / 有风险 / 被阻塞
主动作
```

示例:

```text
定位: Target / Crosshair
受众: Users
素材: Folder / Image
草稿: PenLine
审核: CheckCircle / Eye
发布: Send / Rocket
复盘: ChartLine
```

## Tab 设计规则

主要 tab 不只是文字导航，应有视觉识别:

- 图标 + 1 到 4 个字标签。
- 当前 tab 有明确 active 状态: 底线、背景、色块、图标高亮或进度点。
- 已完成 tab 可显示 check。
- 有问题的 tab 可显示 warning dot。
- 数量默认 3 到 6 个；超过 6 个考虑 sidebar、分组或 more menu。

示例:

```text
[Target] 定位
[Folder] 素材
[PenLine] 草稿
[Eye] 审核
[Send] 发布
[ChartLine] 复盘
```

## 非文字化信息通道

优先使用这些方式传递信息:

- 图标: 表达功能类别。
- 颜色: 表达状态，不表达装饰。
- 形状: 区分对象类型，例如素材、任务、风险、产出。
- 进度: 表达流程位置。
- badge: 表达数量、状态、风险。
- skeleton/loading: 表达生成中。
- preview: 让用户看到结果，而不是解释结果。
- inline validation: 直接标出哪里需要改。
- 空状态插画/示意: 说明下一步，但文字要短。
- hover/focus 动效: 提示可交互。

## 游戏化但不幼稚

可以用:

- 任务进度条。
- 完成勾选。
- 连续步骤高亮。
- 成就 badge。
- 轻微音效或触感反馈，需可关闭。
- 成功动效。
- 风险雷达、能量条、质量分、发布准备度。

不要用:

- 大量 emoji。
- 花哨但无意义的动画。
- 游戏化导致主任务不清。
- 分数看似精确但没有依据。
- 色彩过多导致状态混乱。

## 图标使用约束

- 优先使用项目已有 icon 库；React 项目优先 `lucide-react`，除非已有其他库。
- 图标必须与标签语义一致，不能随机装饰。
- 只靠图标表达时，必须有 `aria-label` 或 tooltip。
- 常见动作使用通用图标: 保存、下载、搜索、过滤、更多、删除、返回、发布。
- 同一概念全项目固定同一个图标。
- 图标尺寸、线宽、颜色和按钮尺寸保持一致。

## 重写检查

交付前检查页面所有可见文字:

```text
这句话能不能压缩成 1 到 4 个字？
这句话能不能变成图标、状态、badge、进度或 tooltip？
这句话是不是 PRD/产品策略/实现说明？
用户不读这句话，能不能仍然知道下一步？
```

如果答案不好，重写 UI。

## 自媒体内容工作台示例

不要:

```text
先把这条内容做给谁、做成什么说清楚
项目页只服务一件事，就是把受众、目标、表达角度、交付和发布时间锁定。
```

改成:

```text
内容定位
```

页面结构:

```text
顶部: 内容定位 + 状态 badge + 主动作「锁定定位」
步骤 tabs:
  [Target] 定位
  [Users] 受众
  [Compass] 角度
  [Package] 交付
  [Calendar] 排期
主区域: 当前 tab 的表单或预览
右侧: 发布准备度 / 缺失项 / 下一步
```

缺失项不写长说明，写成 checklist:

```text
[ ] 受众
[ ] 内容目标
[ ] 表达角度
[ ] 交付形式
[ ] 发布时间
```

## 给 Codex 的执行要求

做前端页面时:

1. 先把流程拆成图标化步骤或 tab。
2. 页面标题不超过 8 个汉字，tab 标签不超过 4 个汉字。
3. 每个主要 tab / step 必须有图标、状态和主动作。
4. 大段解释移出 UI，放到 PRD、tooltip 或帮助抽屉。
5. 先输出“可见文案表”和“图标/状态映射表”，再写代码。
6. 交付前检查是否仍有解释型长句。
