# 可复制到项目 AGENTS.md 的约束片段

以下片段适合放入具体前后端产品项目的 `AGENTS.md`。

```markdown
## 前后端产品闭环约束

本项目所有产品功能、页面、组件和接口必须遵守“产品闭环”原则：用户从入口进入后，应能清楚知道当前在哪里、下一步做什么、点击后发生什么、失败后如何恢复、成功后去哪里。

### 开发前

- 先写用户任务流，再写组件和代码。
- 每个页面只承载一个主目标。
- 每个页面先选择通用产品表面：entry、workflow、management、editor、reader、asset、monitoring 或 settings。
- 每个视图默认只允许一个 primary action。
- 复杂任务必须拆成清晰步骤、分段表单、tabs 或渐进披露。
- UI 上的每个可见动作都必须有对应接口、本地状态或明确的非持久化说明。

### 按钮和状态

- Primary action 推进主流程；secondary/tertiary 不得抢主按钮。
- 删除、发布、扣费、覆盖、AI 批量生成等高风险动作必须有确认、预览、撤销或回滚路径。
- 每个动作必须覆盖 default、hover、focus、disabled、loading、success、error。
- 按钮文案用动词，且同一意图全项目使用同一文案。

### 前端

- 优先复用现有组件、tokens、图标库和布局规则。
- 从零或大改 UI 前，先定义色彩角色、字体层级、布局网格、状态映射、图标语义和素材清单。
- 工作型产品保持安静、清晰、可扫描，避免营销式 hero 和装饰性视觉。
- 不把任何其他项目的真实产品名、页面文案、导航标签、业务对象或品牌风格作为模板复制到本项目。
- 空状态、错误状态、无权限状态必须告诉用户下一步。
- 移动端、键盘操作、focus ring、label、对比度必须检查。
- PRD、需求分析、设计原则、实现策略、验收标准、“我们要/不要怎么做”不得作为用户可见文案出现在前端页面。
- 前端文案只保留用户完成任务所需的标题、标签、动作、状态和短提示；内部说明写入文档或交付说明。

### 后端

- 接口契约必须写清请求、响应、错误码、权限、校验和前端状态映射。
- 长任务必须有 job id、状态轮询或事件流、取消和失败恢复。
- 删除、发布、扣费、覆盖类接口需要幂等或防重复提交。
- AI 功能必须保留人工确认路径，不能直接覆盖用户内容。

### 交付前

- 走通 happy path、空状态、错误状态、加载中、无权限和移动端核心路径。
- 能运行时必须跑 lint/test/build；不能运行时说明原因。
- 列出页面所有可见文案，确认没有 PRD/内部说明泄露。
- 交付说明必须写清闭环了哪条用户动线、覆盖了哪些状态、还有哪些未覆盖风险。
```
