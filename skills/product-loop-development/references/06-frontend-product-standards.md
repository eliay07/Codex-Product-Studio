# 前端成熟产品规范

## 一句话原则

前端不是把需求说明展示出来，而是把用户任务、业务状态和产品判断转译成可操作界面。

## 成熟产品感

- 页面先服务任务，不先展示观点。
- 首屏必须让用户知道：我在哪里、能做什么、下一步是什么。
- 用户不应依赖阅读长段说明理解流程；优先用图标、tab、步骤、状态和视觉层级提示。
- 工作型产品优先清晰、安静、密集、可扫描；不要做营销式大 hero、装饰性卡片堆叠和过度渐变。
- 每个页面只保留一个主目标；其他内容降级为辅助区、折叠区、菜单或详情页。
- 组件尺寸稳定，hover、loading、错误文案不得造成布局跳动。
- 文案、按钮、图标、表格列、表单字段在移动端和桌面端都不能重叠或溢出。

## 信息架构

页面结构优先顺序：

```text
位置感: 标题、面包屑、active nav、步骤条
任务入口: 主动作、输入区、导入区或选择区
任务主体: 表单、列表、编辑器、结果区、预览区
状态反馈: loading、empty、error、success、permission
辅助信息: 历史、说明、设置、筛选、帮助
```

不要把“为什么这样设计”“我们不做什么”“实现方式”放在任务主体。它们属于 PRD、设计说明或交付说明。

## 设计系统

没有现成设计系统时，先建立最小 tokens：

```text
颜色: primary、secondary、accent、background、surface、text、muted、border、success、warning、danger、info
字号: page-title、section-title、body、caption、button
间距: 4、8、12、16、24、32
圆角: 4 或 8，除非项目已有规则
阴影: 少用，只表达层级
动效: 只用于反馈、过渡和状态变化
```

色彩不能只写一组“看起来不错”的 hex。必须定义角色、用途和状态：

```text
色阶: 50 / 100 / 200 / 300 / 400 / 500 / 600 / 700 / 800 / 900
语义: success / warning / danger / info
状态: hover / active / focus / selected / disabled / loading
模式: light / dark，或明确本版本只做 light
比例: 60% 中性底 + 30% 层级和分区 + 10% 品牌强调
```

涉及色彩、版式、视觉舒适度时，先按 `12-color-typography-layout-systems.md` 输出色彩角色表、字体层级表、布局网格表和状态映射表。

禁忌：

- 不用大面积紫蓝渐变、玻璃拟态、发光球、bokeh、装饰 orb 作为默认风格。
- 不让页面被单一色相统治；避免全紫、全蓝、全棕、全米色这类一色到底的界面。
- 不在组件里散落裸 hex；裸色值只允许集中出现在 token 定义文件。
- 不用通用 AI dashboard 套壳：一堆彩色卡片、空洞指标、泛泛 hero、随机 icon。
- 不做卡片套卡片。
- 不把说明文字堆成“教程页”，除非用户正在 onboarding。

## 控件选择

- 工具动作优先用图标按钮加 tooltip：保存、下载、搜索、过滤、更多、撤销、重做。
- 模式选择用 segmented control 或 tabs。
- 主要流程 tab 必须有图标或视觉标识；仅文字 tab 容易显得像文档目录。
- 布尔设置用 switch 或 checkbox。
- 数字范围用 slider、stepper 或 number input。
- 多选/筛选用 checkbox group、combobox 或 menu。
- 危险操作用明确的 destructive 样式，并提供确认、撤销或后果说明。

## 状态覆盖

每个页面至少覆盖：

```text
首次进入
已有数据
空数据
加载中
提交中
成功
失败
无权限
移动端
键盘焦点
```

AI 功能额外覆盖：

```text
等待输入
生成中
部分结果
需要人工确认
已接受
已拒绝
重试中
限流或失败
```

## 交付前视觉检查

- 页面上所有可见文案都像真实产品，不像 PRD 摘录。
- 页面有统一色彩模式和 token，主色、辅助色、状态色职责清楚。
- 版式有稳定网格、对齐和留白，不像随手拼贴。
- 主按钮唯一且显眼。
- 次要动作没有抢主按钮。
- 空状态给出下一步。
- 错误状态给出恢复路径。
- 页面没有无意义的装饰文字、解释型标语或内部策略。
- 浏览器截图中没有明显重叠、截断、错位、空白大洞。
