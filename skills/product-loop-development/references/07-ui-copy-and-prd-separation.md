# UI 文案与 PRD 隔离

## 一句话原则

PRD 负责解释团队怎么想，UI 负责帮助用户完成任务。不要把团队思考过程直接渲染给用户。

## 三类内容

### 可以进入 UI

- 页面标题：当前对象或任务，例如 `素材分析`、`发布设置`。
- 字段标签：用户需要填写或选择的内容。
- 按钮：用户动作，例如 `生成分析`、`保存草稿`、`确认发布`。
- 状态：加载、成功、失败、空状态、无权限。
- 帮助提示：用户完成当前任务所需的短提示。

### 应该进入 PRD/设计文档

- 我们要怎么做、不怎么做。
- 为什么选择这个方向。
- 功能取舍、版本边界、MVP 范围。
- 市场分析、用户研究、设计原则、内部假设。
- 验收标准、技术实现路径、接口计划。

### 应该进入代码注释或提交说明

- 非显而易见的工程原因。
- 数据兼容、性能权衡、迁移策略。
- 临时 hack 和后续清理计划。

## 禁止出现在 UI 的高风险句式

如果这些句子出现在 `.tsx`、`.jsx`、`.vue`、`.html` 的可见文本中，默认要删除或改写：

```text
我们要...
我们不...
不要...
本版本不做...
设计原则...
实现策略...
PRD...
需求分析...
验收标准...
技术方案...
产品闭环...
内部说明...
根据用户要求...
为了避免...
```

例外：这些词出现在内部管理后台、开发者工具、帮助中心或文档编辑器里，且目标用户确实需要看到。

## 改写方式

坏：

```text
我们要避免用户在没有确认的情况下直接覆盖 AI 结果。
```

好：

```text
先预览结果，确认后再覆盖原内容。
```

坏：

```text
本版本不做复杂权限管理，只支持个人空间。
```

好：

```text
当前空间仅你可见。
```

坏：

```text
为了形成产品闭环，请先上传素材再生成分析。
```

好：

```text
上传素材后即可生成分析。
```

更好：如果能用控件表达，就不要写句子。

```text
[Upload] 素材
[Sparkles] 生成
[Eye] 预览
[Check] 确认
```

## 页面文案审查

交付前列出所有可见文案，逐条判断：

```text
这句话是给用户完成任务的吗？
用户需要现在看到它吗？
它有没有暴露内部思考、实现方式或需求文档？
能否改成更短的动作、状态或提示？
```

如果不能通过，移到 `产品闭环计划.md`、`需求文档.md`、`设计说明.md` 或最终交付说明。

## 示例和占位符

规则文档、模板和 prompt 示例必须使用占位符，不使用当前工作区或其他项目里的真实名词：

```text
<ProductName>
<UserRole>
<PrimaryTask>
<Entity>
<PrimaryAction>
<Result>
```

如果需要展示结构，只写“页面标题 / 主动作 / 状态 / 下一步”这类抽象槽位，不写具体产品的页面文案。

## 长句压缩

页面标题、tab、按钮、badge 默认使用短语，不使用完整解释句。

```text
页面标题: 2 到 8 个汉字
Tab 标签: 1 到 4 个汉字
按钮: 动词 + 对象
Badge: 状态词
Tooltip: 一句话，不超过 20 个汉字
```

如果一句话超过 20 个汉字，优先考虑拆成图标、状态、步骤、checklist 或帮助抽屉。
