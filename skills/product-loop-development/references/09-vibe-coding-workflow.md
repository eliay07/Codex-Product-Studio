# Vibe Coding 工作流

## 一句话原则

把 AI 当共同探索产品的搭档：人负责问题、品味、真实行为、减法和验收；AI 负责发散、生成、实现和变体。

## 从 idea 到产品

1. 描述问题，不急着描述方案。
2. 写出真实行为：我现在怎么做、哪里卡、我不会主动做什么。
3. 让 AI 给多个入口和形态，不只给功能列表。
4. 选择最自然的入口：用户已经频繁打开、点击或停留的地方。
5. 先做小而锋利的版本，不做 all-in-one。
6. 自己或目标用户连续用几天，观察是否真的会回来。
7. 删掉没被使用的功能。
8. 把反复有效的流程沉淀成 Skill。

## 和 AI 对话的方式

推荐：

```text
我遇到的问题是...
我真实行为是...
我不会主动...
我希望这个产品改变的行为是...
有哪些入口可以让我不用额外记住去用？
先给我 3 个产品方向，每个方向说明目标用户、主场景、第一版只做什么。
```

不推荐：

```text
帮我做一个功能很多的 AI 工具。
帮我做成熟一点。
帮我做成 SaaS。
```

## 设计探索

- 做 playground：一次生成多组字体、颜色、布局、动画，让人挑。
- 用 `design.md` 描述视觉 token 和方向，不让模型硬套 HTML 模板。
- 给具体视觉参考，不给“高级感”“未来感”这种空词。
- 明确 ban 掉 AI 味：紫蓝渐变、玻璃卡片、空洞 dashboard、无意义 hero、随机 icon。

## 人类负责减法

每轮迭代都问：

```text
这个功能我真的会用吗？
如果没有它，核心任务能不能完成？
它是不是只是 AI 很容易加，所以才被加进来？
它有没有制造新的按钮、状态、接口和维护成本？
```

删掉之后产品更清楚，就删。

## 形成 Skill

不要一开始就写 Skill。先做一轮真实项目：

1. 让 AI 按普通方式做。
2. 第一版通常会暴露问题。
3. 你连续给反馈：保留什么、禁止什么、怎么判断好坏。
4. 项目完成后，让 AI 把这套流程、红线和检查项整理成 Skill。
5. 下一次项目再用 Skill 验证，继续收敛。

## 和 Codex 的分工

- 需要产品 brainstorm、设计探索、审美变体：先用更适合发散和设计的模型或工具。
- 已经知道要做什么，需要读代码、改 repo、跑测试、修 bug、沉淀约束：用 Codex。
- 长期复用的产品判断：不要停留在聊天里，写进 `AGENTS.md`、Skill 或检查脚本。
