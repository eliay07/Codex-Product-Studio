# 产品经理优先级、动线与 2C 体验约束

## 一句话原则

前端不是先画页面，而是先回答“谁最高频使用什么、为什么现在就要用、下一步在哪里、哪些动作该被突出、哪些动作该被隐藏”。

## 参考依据

本文件把产品经理和 UI/UX 常用原则转成 Codex 可执行约束：

- NN/g 的 recognition rather than recall: 界面应让用户识别下一步，而不是让用户读说明或回忆流程。
- NN/g 的 visual hierarchy: 重要元素应通过大小、颜色、对比、分组和位置被优先看到。
- NN/g 的 progressive disclosure: 复杂功能先露出高频主路径，低频和高级选项逐步展开。
- NN/g 的 choice overload / cognitive load: 不要一次性暴露过多同权重选择；用分组、默认推荐、分步任务降低用户解析成本。
- Material Design 的 FAB/primary action 思路: 每个屏幕应聚焦最重要的主动作。
- Material Design 的 layout / usability 思路: 布局要组织元素、表达层级、吸引注意力，并尽量聚焦每页一个主要任务。
- Apple HIG 的 navigation 思路: 使用用户熟悉的导航模式，减少迷路和不必要的复杂交互。
- Apple HIG 的 disclosure controls / sidebars 思路: 当内容很多时，用层级和 disclosure 把细节藏到需要时再展开。

这些原则落到 Codex 时，核心是: 每次改前端代码前，先产出 PM 决策表，再写 UI。

## PM 前置判断

每次从 idea 做产品、改前端、做 2C 页面、官网或工作台时，先填写：

```text
产品定位:
目标用户:
最高频场景:
用户最想完成的第一件事:
用户成功瞬间:
首屏必须看见:
首屏不该出现:
业务目标: activation / retention / conversion / revenue / trust
当前版本不做:
ToC/ToB 取向:
如果是 ToC，创作者成功瞬间是什么:
如果是 ToB，后台管理必要性是什么:
```

如果这些问题答不清，不要先写组件。

## 默认 ToC 取向

除非用户明确说“后台、管理系统、ToB、企业内部、控制台、CRM、ERP、运营后台、管理员工具”，否则默认把原型设计成 ToC 或 prosumer 产品。ToC 原型要优先像“用户想打开来完成一件事的产品”，不是“运营人员管理很多对象的系统”。

ToC 创作者工具默认规则：

- 首屏先给一个大的创作入口，例如主输入、模板入口、灵感卡或立即生成按钮。
- 首屏优先承接创作者当天要做的第一件事：目标进度、可继续的灵感/草稿、一个主创作入口，而不是先展示后台指标。
- 对 AI 生成、调研、需求澄清和产品启动类工具，首屏先让用户输入一句需求或选择示例；问卷、配置、参数和确认项应在用户点击发送/开始后基于输入预填，再以轻量确认流出现。
- 用预览、示例、模板封面、进度、动效和视觉资产引导，而不是用指标卡和说明文字解释。
- 低频功能放到“我的、设置、高级、更多、详情”里，不和主创作路径同级。
- 数据和管理能力要转译成用户语言，例如“作品表现、灵感雷达、下一条建议”，不要默认叫“数据复盘、发布矩阵、素材缺口”。
- 首页优先服务 activation 和 creation，不优先服务 monitoring。

## 创作者产品三段动线

面向个人创作者、自媒体、短视频、图文、播客、课程或作品生产的 ToC / prosumer 工具，默认先按三段动线审阅。除非用户明确要求后台或运营管理，否则不要先做“数据看板 + 功能菜单”。

### 1. 目标管理与灵感库

第一屏要帮助用户开始，而不是管理系统：

- 展示本周期目标、完成进度和可编辑目标入口。
- 提供灵感库，统一管理个人储存灵感、AI 生成灵感、AI 建议和未完成草稿。
- 灵感卡必须能收藏、编辑主题、继续生成或进入草稿。
- 多线程创作场景下，灵感库要允许“先收起来，之后继续”，不要强迫用户一次完成一条内容。
- 首屏只保留一个主动作，例如“开始找灵感”或“新建作品”；目标、收藏、继续编辑作为次动作。

### 2. 沉浸式工作台

用户点击某条灵感、草稿或作品后，应进入围绕单个主题的工作台：

- 中央区域服务核心编辑，例如标题、正文、脚本、分镜或画布。
- AI 能力靠近编辑对象，例如“AI 改标题”“AI 润正文”“生成配音”“生成视频”，不能散落在全局菜单里。
- 素材、配音、视频、导出文件和版本记录围绕同一个项目归档，最好在同一项目目录下有明确路径。
- 高风险动作如覆盖、发布、批量生成必须有预览、确认、撤销或版本回退。
- 工作台默认少文字、多预览、强上下文，让用户清楚当前正在做哪条内容。

### 3. 发布与复盘

发布不是主流程终点，而是复盘闭环入口：

- 发布页提供平台跳转或操作按钮，但平台页完成后必须有“确认已发布”来同步状态。
- 发布记录要保存平台、链接、状态、时间和失败原因。
- 数据追踪优先通过链接、平台记录、手动补录或可接入 API 回流，为后续复盘提供依据。
- 复盘页应把数据、评论信号和下一条灵感连接起来，而不是只展示静态指标。
- 发布和复盘能力属于后段流程，不应抢占创作台第一屏。

ToB / 后台成立条件：

- 目标用户是运营、管理员、销售、客服、财务、团队管理者。
- 主任务是批量管理、审批、监控、配置、报表、权限、工单或异常处理。
- 用户明确需要高信息密度、表格、筛选、批量操作或审计链路。

如果条件不成立，不要把页面默认做成侧边栏 + 指标卡 + 多列管理面板。

## 功能优先级表

把所有功能先排队，再决定导航和页面位置：

```text
功能:
用户任务:
使用频率: high / medium / low
用户价值: high / medium / low
业务价值: high / medium / low
操作风险: high / medium / low
学习成本: high / medium / low
依赖条件:
推荐位置:
动作层级: primary / secondary / tertiary / destructive
```

排序规则：

- 高频 + 高用户价值: 首屏、主导航、默认 tab、主按钮或空状态主入口。
- 高频 + 低风险: 放在用户手边，例如工具栏、列表顶部、底部固定动作区。
- 低频 + 高风险: 放进设置、更多菜单、详情页或确认弹窗，不和主动作挨太近。
- 低频 + 低价值: 延后、不做、折叠或移除。
- 高业务价值但低用户感知: 不能强行抢主流程，应通过恰当时机、状态提示或渐进披露出现。

## 同级页面按钮摆放逻辑

同一级页面的动作位置必须有依据。默认顺序：

```text
导航/返回 -> 上下文信息 -> 核心编辑/查看 -> 辅助工具 -> 主提交动作
```

桌面端常用规则：

- 页面级主动作放在标题区右侧或主任务区末尾。
- 列表页的创建/导入放在列表标题或空状态附近。
- 筛选、搜索、排序靠近列表，不放到页面远处。
- 批量操作只在选中对象后出现。
- 危险动作远离主动作，放在更多菜单、详情底部或危险区域。

移动端常用规则：

- 高频提交动作可放底部 sticky action bar。
- 底部 tab 用于主分区导航，不要塞危险动作或临时操作。
- 表单提交放在用户完成输入后的自然位置；长表单可同时提供底部固定提交。
- 需要连续操作的工具按钮靠近操作对象。

按钮摆放检查：

```text
这个按钮离它影响的对象近吗？
它和同级按钮的视觉权重匹配吗？
用户会不会误点危险动作？
主动作是否只有一个？
低频动作是否已经降级？
```

## 动线模型选择

不要所有产品都硬做成 tab 或卡片。先选择动线模型：

```text
线性向导: 用户必须按顺序完成，例如注册、发布、支付、审批。
Hub and spoke: 用户从中心页进入多个子任务，例如内容工作台、项目首页。
列表-详情: 用户先浏览对象，再进入单个对象操作，例如 CRM、订单、素材库。
画布-工具栏: 用户围绕一个对象连续编辑，例如设计器、编辑器、白板。
搜索-筛选-结果: 用户目标明确，先找对象再行动，例如知识库、商品、文档。
仪表盘-下钻: 用户先看状态，再进入异常或详情，例如运营后台、监控。
探索-选择-确认: 2C 用户需要先被吸引，再选择并完成关键动作。
```

选择规则：

- 有强先后依赖时，用线性向导。
- 用户反复回到同一中心处理多个任务时，用 hub and spoke。
- 对象数量多、需要比较时，用列表-详情。
- 编辑对象复杂、操作密集时，用画布-工具栏。
- 用户知道自己要找什么时，用搜索-筛选-结果。
- 需要看整体健康度时，用仪表盘-下钻。
- 2C 官网或增长页先让用户理解价值，再进入试用或转化。

## 引导式页面逻辑

复杂任务不要一次性展示所有选择。先把任务拆成“一个屏幕一个主要决策”：

```text
1. 开始: 一个主入口或一个推荐集合。
2. 选择: 用户选择模板、灵感、对象或目标。
3. 配置: 只展示当前选择所需的关键参数。
4. 预览: 展示结果、来源、风险和可编辑内容。
5. 确认: 保存、导入、发布或继续编辑。
```

默认不使用“三栏全摊开”承载 ToC 主流程。像“来源 / 候选 / 详情”这类结构，应改为：

- 首屏展示大入口和少量精选推荐。
- 点击推荐后进入详情页、抽屉或步骤页。
- 详情页再提供生成、收藏、导入、编辑等动作。
- 高级来源、筛选、参数放到二级入口或设置面板。

## 可变页面层级

每个产品必须写清楚页面内外层级，避免所有模块同权重。层级深度按功能复杂度决定，不要求每个产品都凑满 5 层；功能少可以 2-3 层，功能多或任务链路长时再扩展到 4-5 层或更深。

```text
基础层: 产品域 / 一级模块 / 主任务入口。
中间层: 步骤、状态、预览、候选、筛选。
深层: 详情、来源、参数、历史、日志、设置、异常和解释。
```

检查规则：

- 一级模块不要超过 5 个，除非用户明确要求后台。
- 每个一级模块只暴露一个默认主任务。
- 不为了满足固定层数而制造空页面、空步骤或无意义入口。
- 功能越少，层级越浅；功能越多，越需要渐进披露和更深的详情层。
- 页面里的卡片不能同等大小、同等颜色、同等按钮权重。
- 详情、来源、参数、日志、数据不应和主入口抢首屏。
- 用户 3 秒内应知道当前位置和下一步。

## 减少文字和信息过载

ToC 和创作者工具必须减少文字堆砌：

- 用标题 + 视觉预览 + 一个动作表达主价值，避免大段说明块。
- 用图标、封面、缩略图、状态 badge、进度条和示例结果代替长解释。
- 一屏最多强调 1 个主动作、2-3 个次级选择、1 个恢复路径。
- 候选内容用卡片、封面、标签和一句话差异表达，不要让用户读完整说明后才知道区别。
- 如果需要解释，放在 tooltip、详情页、折叠面板或帮助入口。

反例：

- 首页全是指标色块和说明文字。
- 选题页同时展示选题来源、候选列表、选题详情、参数和生成结果。
- 所有按钮都是同一颜色，用户不知道先点哪个。
- 把内部系统状态、数据库、真实 AI、本地预览等工程概念暴露给普通用户。

## 产品表面选择

动线模型决定用户如何前进，产品表面决定页面如何承载任务。每次改前端前，先选择一个主表面：

```text
Entry surface: 用户开始最高频任务。
Workflow surface: 用户跟随步骤完成任务。
Management surface: 用户管理多个对象。
Editor surface: 用户编辑一个对象。
Reader surface: 用户阅读、审阅或导出结果。
Asset surface: 用户浏览、筛选和复用资产。
Monitoring surface: 用户查看状态、异常和趋势。
Settings surface: 用户配置低频或高风险选项。
```

如果一个页面同时承担多个表面，优先拆分；不能拆分时，必须指定主表面，并把其他表面降级为侧栏、抽屉、详情页或菜单。

## 工作台交互层级

创作者工具、AI 工具和编辑器产品如果需要承载多个能力，不要把功能全部铺到一个页面。先定义工作台层级：

```text
入口层: 用户从目标、灵感、最近记录或模板开始。
对象层: 用户选择一条草稿、作品、素材、会话或项目。
编辑层: 用户确认打开后进入单对象编辑区。
检查层: 来源、参数、属性、版本、风险和预览进入 inspector 或 drawer。
运行层: 生成过程、日志、错误、上传队列进入 bottom panel。
快捷层: 全局搜索、命令、低频设置进入 command palette 或 floating action。
恢复层: 所有被收起的 sidebar/panel/drawer 都要有持续可见的恢复入口。
```

默认态先展示入口层和对象层，不自动替用户进入编辑层。编辑层必须由“打开、继续、确认、创建后打开”等明确动作触发。

折叠规则：

- 收起不是删除。收起后必须保留 rail、handle、tab、floating trigger 或命令入口。
- 展开时优先恢复上一次尺寸或状态，不要只恢复到最小值。
- 任何遮罩、drawer 或浮层不能让用户失去导航、关闭或恢复路径。
- 多面板移动端必须降级为 stepper、bottom sheet、full-screen detail 或单列工作流。

## 2C 首屏规则

2C 产品首屏必须服务“立刻理解 + 立刻行动”。首屏优先放：

```text
产品是什么: 1 句短价值表达
给谁用: 目标人群或场景信号
最高频动作: 输入框、上传、开始、试用、生成、搜索等
结果预期: 示例、预览、真实截图或可交互 demo
信任信号: 数据来源、用户反馈、安全/隐私、品牌背书，按需出现
```

首屏不要放：

- 大段理念。
- 内部产品策略。
- 功能大全。
- 低频设置。
- 还没有输入需求就出现的完整问卷、参数表和方法论清单。
- 空洞 slogan。
- 需要读很多字才懂的流程说明。

2C 首屏判断：

```text
用户 3 秒内知道这是干什么的吗？
用户 5 秒内知道能点哪里开始吗？
首屏展示的是最高频/最吸引人的动作吗？
视觉是否让用户看到结果，而不是读功能说明？
问卷和确认项是否后置到发送/开始之后？
```

## UI/UX 视觉定位

视觉不能先选好看的颜色，要先由产品定位推导：

```text
目标用户: 新手 / 专业人士 / 创作者 / 消费者 / 企业客户
使用场景: 高频工作 / 轻娱乐 / 决策购买 / 创作编辑 / 数据监控
情绪目标: 安全 / 高效 / 灵感 / 信任 / 兴奋 / 克制
视觉性格: 专业 / 温暖 / 游戏化 / 科技 / 高级 / 活泼
信息密度: 高 / 中 / 低
动效强度: 无 / 轻 / 中 / 强
```

配色选择：

- 工作台/后台: 中性底、清晰状态色、低饱和品牌色。
- 2C 创意工具: 可有更强主色、品牌背景、插画、模板封面、预览卡和动效反馈，但仍要克制。
- 金融/健康/法律: 稳定、可信、对比清楚，避免花哨。
- 游戏化产品: 用进度、成就、反馈和图标表达，不靠彩虹色堆叠。
- 高端官网: 少色、强留白、真实素材、清晰排版和精确动效。

## 创作者工具视觉规范

创作者工具不能只像后台管理台。默认应该呈现为“灵感 -> 创作 -> 作品”的产品体验：

- 首屏使用视觉锚点：生成输入框、模板封面、灵感卡、作品预览或 GPT 生成背景图。
- 列表要有封面、缩略图、状态、推荐理由和使用入口，不只是一行文字。
- 结果页要有预览优先的结构，编辑、来源、参数降级到侧栏或二级 tab。
- 空状态要给样例和开始按钮，不只写“暂无数据”。
- 成功态要展示产出物、下一步和分享/继续创作入口。
- 后台能力保留，但包进“我的、设置、高级、运营”而不是抢主导航。

## 素材库计划

需要官网、2C 页面、品牌化产品或视觉改造时，先规划素材库：

```text
logo / wordmark:
app icon:
favicon:
hero background:
feature icons:
empty state illustration:
onboarding illustration:
social preview image:
loading / success / error visuals:
product screenshots or mockups:
texture / pattern / background system:
```

生成方式必须提前声明：

```text
GPT 图像生成: hero background / feature illustration / empty state / social preview / cover / campaign visual / template thumbnail
代码原生资产: icon / simple SVG mark / data diagram / deterministic UI mock
真实产品素材: screenshot / product capture / uploaded user asset
```

素材库要求：

- 所有素材共享同一视觉语言、色卡、线条、圆角、光影和构图。
- 官网背景图、logo、图标、空状态和社交图不能各像一个不同项目。
- 能用真实产品截图或可交互预览时，不用抽象装饰图替代。
- 背景图、插画、社交预览图、封面图、模板缩略图等品牌视觉资产，优先调用 GPT 图像生成产出 bitmap，再保存到项目资产目录；不要用 HTML/CSS/SVG 临时画几根线来冒充完整视觉资产。
- AI 生成素材前先写风格 brief、尺寸、用途、文件名和版权/可用性说明。
- 素材应落到明确目录，例如 `src/assets/brand/`、`public/brand/` 或项目现有资产目录。
