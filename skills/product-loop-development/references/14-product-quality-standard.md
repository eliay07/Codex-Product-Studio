# 通用产品质量标准

## 一句话原则

不要把任何单个产品、单个行业或单个页面当成固定模板。成熟产品来自清晰的用户任务、合适的信息架构、稳定的视觉系统、完整的状态反馈和可验证的前后端契约。

默认判断：用户没有明确要求 ToB、后台、企业管理系统或内部运营工具时，按 ToC / prosumer 产品质量标准设计。原型要像用户愿意打开并完成创作、消费、购买、学习或表达的产品，而不是默认像管理后台。

## 禁止污染提示词

插件规则、模板、示例和检查脚本不得写入具体项目名、真实品牌名、真实用户姓名、真实业务对象或其他项目里的页面名称。

示例必须使用占位符：

```text
<ProductName>
<UserRole>
<PrimaryTask>
<Entity>
<PrimaryAction>
<Source>
<Result>
```

可以写产品类型，不能写具体产品案例。可以写结构规则，不能把某个已有产品的导航、文案、配色或业务对象原样固化。

## 产品质量门槛

每次生成、改造或评审产品体验时，先建立这 8 个门槛：

```text
1. 用户门槛: 目标用户具体，且不是泛泛的“用户”；未明确 ToB 时默认按 ToC / prosumer 判断。
2. 任务门槛: 首屏或默认页暴露最高频、最高价值的第一动作。
3. 动线门槛: 用户知道当前位置、下一步、成功后去向和失败后恢复路径。
4. 结构门槛: 页面结构匹配产品表面，不把所有场景都做成 dashboard 或 landing page。
5. 动作门槛: 每个视图默认只有一个 primary action，低频和高风险动作降级。
6. 状态门槛: loading、empty、error、disabled、success、permission 至少覆盖核心路径。
7. 视觉门槛: 有 token、排版层级、网格、状态色、组件状态和素材系统；ToC 创作者产品必须有视觉锚点、预览或品牌资产。
8. 契约门槛: UI 可见动作对应 API、本地状态、权限、错误码和验收方式。
9. 分层门槛: 按功能复杂度写清页面层级，不把来源、候选、详情、参数、数据同屏同权重堆叠；功能少不硬凑深层级，功能多才加深。
10. 交互架构门槛: 工作台、编辑器和复杂工具必须定义默认态、选择态、编辑态、专注态、检查器、底部面板、浮动工具和可恢复折叠；任何收起状态都不能让恢复入口消失。
11. 后置确认门槛: AI 产品、生成类工具、调研工具和产品启动工具，首屏只承接输入/开始；问卷、配置、参数和确认项必须在用户发送/开始后基于输入预填并逐步确认。
```

任何一个门槛缺失，都先补齐设计或产品判断，再写页面。

## 通用产品表面

先选择产品表面，再决定布局和控件。一个产品可以组合多个表面，但每个页面只承担一个主目的。

| 表面 | 适用任务 | 常见结构 | 主要风险 |
|---|---|---|---|
| Entry surface | 让用户开始最高频任务 | 品牌/位置、短价值表达、主输入或主动作、示例或预览、信任信号；需要确认时发送后再弹出 | 变成营销说明页，主动作不清；把完整问卷默认铺在首屏 |
| Workflow surface | 处理连续步骤或长任务 | 顶栏、步骤/进度、主工作区、状态流、右侧辅助区 | 过程不可见，失败无法恢复 |
| Management surface | 管理对象集合 | 标题动作区、搜索筛选、列表/表格、批量操作、详情入口 | 指标和卡片堆砌，操作离对象太远 |
| Editor surface | 围绕一个对象持续编辑 | 对象标题、画布/编辑区、工具栏、属性面板、预览/历史 | 工具太散，保存/撤销状态不明确 |
| Reader surface | 阅读、审阅、批注或导出结果 | 目录/进度、内容主体、引用/来源、批注动作、导出 | 信息太长不可扫描，来源不可追溯 |
| Asset surface | 浏览、复用和沉淀资产 | 分类/筛选、网格/列表、预览、标签、收藏/导入动作 | 资产无语义分类，空状态无下一步 |
| Monitoring surface | 查看状态、异常和趋势 | 状态摘要、趋势图、异常列表、告警、下钻入口 | 虚假精确指标，异常没有处理路径 |
| Settings surface | 配置低频或高风险选项 | 分组表单、说明、权限、危险区域、保存/取消 | 设置抢主流程，高风险动作无确认 |

## ToC 创作者产品表面

当产品面向个人创作者、消费者、学习者、求职者、内容生产者或泛用户时，优先使用 ToC / prosumer 表面，而不是后台表面：

| 表面 | 适用任务 | 常见结构 | 主要风险 |
|---|---|---|---|
| Creation entry | 让用户立刻开始创作 | 大输入、模板入口、灵感卡、示例预览、一个主按钮；确认项后置 | 变成指标首页、功能目录或问卷首页 |
| Goal hub | 管理个人创作目标和进度 | 周期目标、进度条、可编辑目标、继续创作入口 | 变成 KPI 看板，目标不能修改 |
| Guided workflow | 分步完成复杂生成 | 步骤、进度、当前选择、预览、下一步 | 一页堆满来源、参数和详情 |
| Inspiration feed | 浏览和选择灵感 | 封面卡、标签、推荐理由、收藏/使用 | 变成纯文本列表 |
| Immersive workspace | 围绕单条内容持续创作 | 中央编辑区、AI 辅助、素材/配音/视频侧栏、项目路径 | 工具散落，文件与内容脱节 |
| Result preview | 确认产出物 | 结果画布、编辑入口、版本、导出/发布 | 来源和设置抢占预览 |
| Personal library | 管理个人作品 | 作品封面、状态、最近编辑、继续创作 | 变成后台表格 |
| Publish cockpit | 发布确认和平台状态同步 | 平台按钮、确认已发布、链接、失败恢复 | 把平台发布误认为已完成，无人工确认 |
| Review loop | 从数据和评论回到下一条内容 | 链接追踪、数据摘要、评论信号、下一条灵感 | 只做静态报表，不产生新动作 |

后台表面只有在用户明确要求管理、监控、批量处理、权限、报表或内部运营时才作为默认。

## 工作台交互质量门槛

复杂工具、创作者工作台和编辑器交付前必须回答：

```text
1. 默认态: 是否只展示入口、最近记录或对象选择，而不是自动打开深层编辑？
2. 选择态: 用户点击某个对象后，是进入预览/摘要/检查器，还是明确打开编辑？
3. 编辑态: 主编辑区是否占最大面积，工具是否靠近被编辑对象？
4. 检查器: 来源、参数、版本、风险、详情是否进入 drawer/inspector，而不是挤占默认主区？
5. 运行面板: 日志、生成过程、错误和上传队列是否进入 bottom panel，并能关闭/恢复？
6. 折叠恢复: 每个 sidebar、list、panel、drawer 收起后是否有持续可见 rail/trigger/handle？
7. 状态记忆: 展开时是否恢复上一次宽度、高度或打开状态？
8. 浮动入口: AI 助手、快捷命令或帮助是否不遮挡主按钮和正文？
9. 命令入口: 低频动作是否进入 command palette、更多菜单或快捷搜索，而不是铺成按钮墙？
10. 移动端: 多面板是否降级成步骤、bottom sheet、full-screen detail 或单列流程？
```

## 创作者三段闭环质量门槛

个人创作者工具交付前必须回答：

```text
1. 目标管理: 用户能否看到并修改本周期目标？完成进度是否来自真实草稿/发布状态？
2. 灵感库: 个人灵感、AI 生成灵感、AI 建议和未完成草稿是否能收藏、编辑、继续？
3. 单项目归档: 某个主题下的草稿、配音、视频、封面、发布链接和复盘数据是否有共同项目或目录？
4. 沉浸编辑: 点击一条记录后，是否进入围绕单条内容的编辑工作台，而不是继续停留在列表页？
5. AI 辅助: AI 改标题、润正文、生成配音、生成视频等动作是否靠近编辑对象，并覆盖 loading/success/error？
6. 发布确认: 平台跳转或外部发布后，是否有人工确认同步状态，而不是自动假设已发布？
7. 数据回流: 发布链接、平台状态、评论信号和复盘结果是否能变成下一条灵感或行动？
```

## 页面骨架要求

每个页面在实现前先写出：

```text
页面目的:
选择的产品表面:
主用户任务:
ToC/ToB 取向:
页面层级:
主动作:
次动作:
高风险动作:
核心状态:
空状态下一步:
错误恢复路径:
移动端主路径:
```

## 渐进披露要求

复杂页面在默认态只呈现一个主入口或一个主决策。详情、来源、参数、日志、数据、设置必须按需进入：

```text
默认层: 主入口 / 推荐 / 预览 / 主动作
中间层: 选择列表 / 筛选 / 模板 / 候选 / 步骤配置
深层: 详情页 / 抽屉 / 来源 / 参数 / 历史 / 异常 / 高级设置 / 系统配置 / 调试信息
```

如果一个页面看起来像“多个卡片和按钮平均分配”，先重新建立层级，再写 UI。

AI 生成、调研、需求澄清和产品启动工具必须优先使用“输入 -> 发送 -> 预填确认 -> 工作台”的结构。首屏不应直接展示完整问卷；问卷只在用户已经表达需求后出现，并且要允许用户确认、修改、补充或跳过非关键项。

## 视觉系统要求

从零做或大改前端时，必须先产出：

```text
色彩角色表: token / 色值 / 用途 / 限制
字体层级表: token / size / line-height / weight / 用途
布局网格表: page padding / gap / max-width / columns / breakpoints
组件状态表: default / hover / focus / active / disabled / loading / success / error
图标语义表: 概念 / 图标 / 使用位置 / aria-label 或 tooltip
素材清单: logo / favicon / empty state / loading / success / error / preview / social image
GPT 图像生成资产: hero background / illustration / cover / template thumbnail / social preview
```

设计方向必须由目标用户和使用场景推导，不允许只写“高级”“科技感”“好看”。视觉风格可以变化，但 token、层级、状态和响应式规则必须稳定。

## 文案要求

可见文案只服务用户任务：

```text
页面标题: 对象或任务名
按钮: 动词 + 对象
Tab: 1 到 4 个字或短词
Badge: 状态词
Tooltip: 当前动作的短说明
空状态: 当前为空 + 下一步
错误状态: 发生什么 + 怎么恢复
```

不要在 UI 写：

```text
产品策略
设计原则
实现路径
需求分析
验收标准
团队取舍
当前版本不做什么
```

## 验收方式

交付前至少验证：

```text
1. 主路径可完整完成。
2. 空、错、加载、禁用、无权限都有界面。
3. 主动作唯一且位置合理。
4. 搜索、筛选、排序靠近被影响的内容。
5. 高风险动作远离主动作，并有确认、撤销或回滚。
6. 页面没有散落裸色值，除 token/theme 文件外。
7. 页面没有 PRD 或内部说明泄露。
8. 移动端核心任务可完成。
9. 键盘焦点、label、aria-label 和对比度可接受。
10. ToC 原型不是后台默认样式，首屏有创作入口、视觉引导和可预期结果。
11. 页面层级深度与功能复杂度匹配，细节和高级能力没有抢首屏。
12. AI/生成/调研/产品启动类首屏没有默认平铺完整问卷，问卷或确认流在发送/开始后出现。
13. lint/test/build 或等价手动验证已运行并记录。
```

## 复用约束

当你从已有产品里学习经验时，只能抽象成：

```text
任务类型
结构模式
状态处理
组件关系
视觉 token 方法
验收检查项
```

不能复用：

```text
具体名称
具体业务对象
具体页面文案
具体导航标签
具体品牌风格
具体数据指标
```
